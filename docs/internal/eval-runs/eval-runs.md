### Toolkit findings from this batch (sensor readings 8-13, continuing batch 1's numbering)

8. **THE BATCH'S HEADLINE META-READING: advisory findings must be ground-truth-verified before they become calibration input.** R7 (Sonnet, HIGH effort) confidently triaged all 11 deanpeters U6 errors as "checker false positives" because the linked files exist relative to the REPO ROOT, and recommended weakening U6 to resolve against the repo tree. That recommendation is WRONG: CommonMark/GitHub resolve relative links against the CONTAINING FILE's directory, so `[problem-statement](skills/problem-statement/SKILL.md)` inside `skills/lean-ux-canvas/SKILL.md` genuinely 404s when clicked (verified by hand against the file). U6 (reference-links) flagged real navigation defects; the plugin-side remediation (normalize to `../`) is right, the toolkit-side one is not. Had this been applied as a calibration it would have blinded U6 to a real defect class. **Disposition: adopted as a METHODOLOGY rule (verify before calibrate); also a small U6 message improvement candidate - say "resolves relative to the containing file" in the finding text so a reader (or a reviewing model) is not misled the same way.**
9. **S7 (command-contract) assumes one-to-one `maps-to`; real plugins ship one-to-many orchestrator commands.** deanpeters' 6 commands each declare a `uses:` list of multiple skills by design (R7), and phuryn's commands map by naming convention (R8). Under `plain-plugin` S7 is already off (house provenance), so third-party grading is unaffected; the reading is for the Standard's own evolution: a future S7 revision could accept `uses: [...]` as an alternative contract form. **Disposition: Standard-evolution note, no urgency.**
10. **U5 (description scorer) cross-corpus evidence is now systematic.** lenny: 50 of 86 descriptions at exactly 0.65, traced to the uniform CONTRIBUTING-template stem "Help users [verb]..." on descriptions whose trigger clauses are actually strong and disjoint (R6 read a dozen). deanpeters: 12 borderline warnings penalizing deliberate brevity (their 200-char web-upload constraint). phuryn: 0.55-0.65 across skills. Combined with batch 1's Anthropic-skills 0.65 cluster and the 12-verb regex trace, the scorer demonstrably under-scores at least three distinct GOOD description styles. **Disposition: the U5 recalibration (ADR 0029 follow-up) now has a real multi-corpus TDD set; this should be the next calibration shipped.**
11. **Cross-skill trigger-surface redundancy is invisible to the gate and is the advisory layer's most distinctive value-add.** R6 found the lenny ai-evals / building-with-llms overlap (two skills satisfying the same query about equally); R7 found the deanpeters workshop pairs (create-a-map vs run-a-workshop descriptions that route ambiguously). U5 scores each description in isolation; no deterministic check compares trigger surfaces ACROSS skills. **Disposition: candidate advisory-report dimension (a standard "trigger-surface collision" section in review mode); a deterministic version (description similarity) is possible but needs careful design to avoid false positives - record for the eval-engine backlog.**
12. **Manifest-vs-disk skill drift is deterministically checkable and the gate misses it today.** R7's one major finding - marketplace.json registering 47 of 49 on-disk skills, making the two newest skills invisible to installers - required no judgment, only comparing the manifest list to the `skills/` directory listing. askit checks its OWN index drift at Gold (G4 index-drift) but has no objective-tier check for a third-party plugin's marketplace/plugin manifest vs disk. **Disposition: NEW-CHECK CANDIDATE (objective provenance) - needs an ADR since it touches the spine; high value for outward grading (it catches silent delivery failures).**
13. **Haiku effort sensitivity: the low-to-medium step is high-leverage.** R8 (Haiku/medium) produced veracity spot-checks, command-completeness analysis, and marketplace-scope reasoning that R3 (Haiku/low) did not attempt, at ~40% more tokens. For scale runs where Haiku is the budget choice, instruct medium effort by default. **Disposition: dossier guidance note (applied).**
14. **First same-target A/B (R8 vs R9): a cheap model's "verified" claim is not verification.** On the identical pm-toolkit target, R8 (Haiku/medium, 54.7k) reported "no veracity gaps detected" and even praised the resume skill's sourcing; R9 (Opus/high, 100.3k, ~1.8x cost) hand-verified the content and found three externally-checkable knowledge errors R8 missed - 'VIPA' cited as a US privacy statute (Virginia's law is the VCDPA), an inverted American-English 'data' agreement rule taught as fact in the grammar skill, and British collective-noun usage flagged as an error - plus the review-resume command's 10-point scorecard contradicting the skill's own 10-point list (only one item overlaps). Both runs agreed on the structural read (clean, coherent, well-disclaimed). Implication: model tier changes WHAT KIND of finding is reachable, not just depth - knowledge-content review (factual, legal, domain claims) needs Opus/high or equivalent; Haiku-tier output should never be treated as a veracity check. **Disposition: METHODOLOGY guidance (applied to the quality-assessment section's evidence base) + dossier lesson.**
15. **A command's enumerated content vs its mapped skill is mechanically comparable, and nothing checks it.** R9's command-vs-skill contradiction (two divergent canonical "10 best practices" lists between `commands/review-resume.md` and `skills/review-resume/SKILL.md`) is detectable without judgment in its strong form: when a command declares a mapping (`maps-to`/`uses:`) and both files enumerate same-cardinality named lists, divergence is a lint. Needs design care (when are lists "the same list"?) but the failure mode - a command whose output contradicts its stated source of truth - is high-value. **Disposition: eval-engine backlog note; possible future S-tier check or review-mode standard dimension.**

**Process notes.** R6 demonstrated the sampling protocol for collection-scale targets (deep-read a stratified sample including name-collision pairs, grep the rest for claim consistency) - worth standardizing in the pipeline (see backlog E11, the dependable eval-run pipeline). R8's raw output contained an em-dash that had to be stripped per house style when transcribing the advisory JSON; the dispatch prompt's plain-ASCII instruction is not fully reliable on Haiku.

### Toolkit findings from this batch (sensor readings, with proposed disposition)

1. **`evaluate.mjs` component scope silently ignores `--profile`.** Found pre-run: `evaluate.mjs <skill-dir> --profile plain-plugin --json` returns `profile: undefined` and house checks (U5) still fire on a third-party single skill. Plugin scope honors the flag; component scope drops it without a warning. Either apply profiles in component scope or reject the flag there; silent acceptance is the worst option. **Disposition: calibration candidate (ADR + TDD).**
2. **The behavioral mode's `evals/` convention has zero instances anywhere.** Neither the toolkit's 23 skills nor any third-party target ships an `evals/` eval-set; both behavioral runs had to derive ad-hoc case sets. The mode is currently aspirational, and the derive-from-description fallback both graders used is undocumented. Either document the fallback in `askit-evaluate` behavioral mode and `agents/askit-quality-grader.md`, or ship eval-set fixtures (askit-build-samples exists to author them). **Disposition: doc/spec fix candidate.** (Also surfaced independently by R1 as an info finding.)
3. **`agents/askit-reviewer.md` step 2 names `check.mjs` where the targeted `evaluate.mjs` is the right tool** (R1's minor finding, independently confirmed: `check.mjs` has no component scope and grades a bare skill directory as an empty plugin root, U8 "No skills found under skills/", vacuous pass). The reviewer delegate should run `node scripts/evaluate.mjs <target> --json`. **Disposition: doc fix candidate.**
4. **U6 residual false-positive class confirmed on a second corpus plugin.** R2 confirmed both `documentation-pipeline` U6 errors are the known Finding-5 residual class (bare example links in NO_CODE_EXAMPLES prose: `[Component Library](link)`, `[docker-compose.yml](../docker-compose.yml)`). Strengthens the ADR 0032 residual dataset; still deliberately not auto-suppressed (the same class hides genuine link-rot). **Disposition: data point for the existing Finding-5 residual decision; no action now.**
5. **U5 scorer corpus data for the deferred recalibration.** Both Anthropic skills (mcp-builder, pdf) cluster at exactly 0.65; R3 traced why: the scorer's action-verb regex is a strict 12-verb list, so 'Guide for creating...' and gerunds ('creating', 'building') miss it. Direct input for the deferred ADR 0029 follow-up. **Disposition: feeds the U5 recalibration TDD set.**
6. **`askit-build-skill` chain-vs-procedure gap (R4 case B4, a real fail).** The frontmatter `chain` declares `askit-skill-author` delegation and the Scope section asserts it, but neither create nor improve mode steps ever instruct the delegation - a literal execution does the work inline. Add an explicit delegate step or state the subagent is an optional implementation detail. **Disposition: skill improvement candidate.**
7. **Behavioral summary semantics are underspecified.** The `summary` block (`fired`/`missed`/`behaviorPass`/`behaviorFail`) has no documented rule for where a failed *no-fire* trigger case counts (R4 filed its T7 no-fire fail under `behaviorFail`); the schema in `docs/reference/evaluation-reports.md` names the fields but not their semantics. Define: `fired` = should-fire cases that fire, `missed` = should-fire cases that do not, and add a false-fire counter or fold no-fire fails explicitly. **Disposition: doc/schema fix candidate.**
