# plugin-eval - profile

plugin-eval is a three-layer quality evaluation framework for Claude Code plugins and skills, living as one plugin inside the wshobson/agents multi-harness marketplace (MIT license, Seth Hobson). It provides a Python CLI (`uv run plugin-eval score <path>`) and Claude Code slash commands (`/eval`, `/certify`, `/compare`) that grade individual skill directories (or whole plugin directories in static-only mode) across 10 weighted quality dimensions using three layers: deterministic static analysis (instant, free), an LLM judge (4 async Claude calls, ~30 s), and Monte Carlo simulation (50-100 LLM calls, ~2-5 min). Each combination of layers maps to a named confidence label (Estimated/Assessed/Certified/Certified+) and a badge tier (Bronze/Silver/Gold/Platinum) derived from a composite 0-100 score. The broader repo ships 84 plugins to six harnesses (Claude Code native plus five generated - Codex CLI, Cursor, OpenCode, Gemini CLI, and GitHub Copilot) from one Markdown source via `make generate HARNESS=<h>`.

## Dimensions

1. **Unit of evaluation:** per-skill for the full three-layer evaluation; per-plugin (static layer only, confidence label forced to "Estimated") when a whole plugin directory is pointed at. The CLI `_detect_target` function explicitly routes to `evaluate_skill` or `evaluate_plugin` based on whether a `SKILL.md` or `.claude-plugin/` marker is found. Plugin-level evaluation bypasses the LLM judge and Monte Carlo layers entirely regardless of `--depth` argument, and the CLI prints a warning to that effect. The unit is therefore per-skill for any statistically calibrated result. [source](https://github.com/wshobson/agents/blob/main/plugins/plugin-eval/src/plugin_eval/cli.py) and [source](https://github.com/wshobson/agents/blob/main/plugins/plugin-eval/src/plugin_eval/engine.py) (Confidence: High)

2. **Tiered and climbable:** score with named confidence labels and badge tiers - the three layers are not a climbable gate but an additive pipeline: running more layers increases the composite score's confidence label (Estimated -> Assessed -> Certified -> Certified+). The score itself is a continuous 0-100 composite. Badge tiers (Bronze >= 60, Silver >= 70, Gold >= 80, Platinum >= 90; Elo thresholds also apply when corpus data is available) are derived from the score. There is no burndown checklist; a skill improves its score by improving quality dimensions, not by satisfying discrete checkboxes. [source](https://github.com/wshobson/agents/blob/main/plugins/plugin-eval/src/plugin_eval/models.py) and [source](https://github.com/wshobson/agents/blob/main/docs/plugin-eval.md) (Confidence: High)

3. **Verdict basis:** hybrid - Layer 1 (static) is fully deterministic (pure Python, no LLM calls, < 2 s): structural checks, frontmatter quality, anti-pattern detection, harness-portability lint. Layer 2 (LLM judge) uses Claude (Haiku + Sonnet) for four semantic dimensions - triggering accuracy (F1 over 10 synthetic prompts), orchestration fitness, output quality, and scope calibration - each with anchored 5-point rubrics. Layer 3 (Monte Carlo) generates 15 varied prompts via Claude Haiku and runs N simulations to measure activation rate, output consistency, failure rate, and token efficiency with statistical CIs. The three layers' scores are blended per dimension using fixed weights into the composite. [source](https://github.com/wshobson/agents/blob/main/plugins/plugin-eval/src/plugin_eval/layers/judge.py) and [source](https://github.com/wshobson/agents/blob/main/docs/plugin-eval.md) (Confidence: High)

4. **Lifecycle coverage:** evaluate + certify; no author scaffolding, no version enforcement, no deprecation or governance stage. The framework grades existing skills and issues badges; it does not scaffold skill authoring, enforce semver on skill metadata, or define a deprecation lifecycle. The `init` command builds a corpus index for Elo ranking but this is evaluation infrastructure, not lifecycle management. [source](https://github.com/wshobson/agents/blob/main/plugins/plugin-eval/README.md) and [source](https://github.com/wshobson/agents/blob/main/docs/plugin-eval.md) (Confidence: High)

5. **Cross-agent emission:** multi-format native via the parent repo's adapter pipeline; plugin-eval itself only grades, it does not emit. The repo-level Makefile defines `HARNESSES := codex copilot cursor gemini opencode` and `make generate HARNESS=<h>` invokes `tools/generate.py` to produce harness-native artifacts for all five. The validate.yml CI workflow runs `make generate-all` and asserts no drift. The `harness_portability` static sub-check (6% weight) grades skills against cross-harness constraints (Codex 8 KB cap, CamelCase tool name refs, bare model aliases, built-in agent name collisions). [source](https://github.com/wshobson/agents/blob/main/Makefile) and [source](https://github.com/wshobson/agents/blob/main/.github/workflows/validate.yml) (Confidence: High)

6. **Self-proving / dogfooded:** partially. The CI `validate.yml` runs plugin-eval's own test suite (`uv run pytest`) on every PR and push, so the framework is tested against itself. The `eval-report.yml` CI workflow runs `eval_all.py` (the batch sweep using `plugin_eval.engine.EvalEngine` directly) on a weekly schedule against all plugins in the repo, including plugin-eval itself. However, there is no committed gate that hard-blocks a merge if plugin-eval's own composite score falls below a threshold; the eval-report workflow runs with `if: always()` and uploads results as artifacts rather than failing the build. [source](https://github.com/wshobson/agents/blob/main/.github/workflows/validate.yml) and [source](https://github.com/wshobson/agents/blob/main/.github/workflows/eval-report.yml) (Confidence: High)

7. **Target spec(s):** Claude Code (SKILL.md + `.claude-plugin/plugin.json` as source of truth); harness portability layer also checks Codex CLI constraints (8 KB body cap, built-in agent name collisions). No reference to agentskills.io, Gemini, or OpenCode spec documents in the evaluation rubric source; portability is graded by detecting Claude-specific patterns that are non-portable, not by validating against a Gemini or OpenCode schema. [source](https://github.com/wshobson/agents/blob/main/plugins/plugin-eval/src/plugin_eval/layers/static.py) and [source](https://github.com/wshobson/agents/blob/main/plugins/plugin-eval/src/plugin_eval/layers/harness_portability.py) (Confidence: High)

8. **Validation depth:** frontmatter-quality + refs + harness-portability + strict-CI. Static layer checks: frontmatter shape (name, description length, trigger-phrase presence), orchestration wiring (output/input docs, code examples), progressive disclosure (line count 200-600 sweet spot, references/ and assets/ dirs), structural completeness (heading density, code blocks, examples/troubleshooting sections), token efficiency (MUST/NEVER/ALWAYS density, duplicate-line detection), ecosystem coherence (cross-refs to related skills/agents), and harness portability (Codex 8 KB cap, CamelCase tool refs, bare model aliases, built-in agent name collisions). Anti-pattern detection adds a multiplicative penalty (max(0.5, 1.0 - 0.05 * count)). The `ORPHAN_REFERENCE` anti-pattern flags dead links to files declared in references/. validate.yml enforces JSON validity of all plugin.json/marketplace.json files and asserts generated harness artifacts match `make generate-all` on every PR. [source](https://github.com/wshobson/agents/blob/main/plugins/plugin-eval/src/plugin_eval/layers/static.py) and [source](https://github.com/wshobson/agents/blob/main/.github/workflows/validate.yml) (Confidence: High)

9. **Versioning support:** none for skills. The plugin-eval package itself is versioned (`version = "0.1.0"` in pyproject.toml). Individual skills have no `version` field in their SKILL.md frontmatter that the parser reads or the static analyzer scores; the parser extracts frontmatter for name and description only. No semver enforcement, contract-lock, or provenance trail for skill metadata. [source](https://github.com/wshobson/agents/blob/main/plugins/plugin-eval/pyproject.toml) and [source](https://github.com/wshobson/agents/blob/main/plugins/plugin-eval/src/plugin_eval/parser.py) (Confidence: Medium - inferred from the parser source; absence of a version field check is absence of evidence for enforcement, not confirmation that no version field exists in any SKILL.md frontmatter)

10. **Eval / test scaffolding:** yes, comprehensive. The test suite covers 15 test files (test_static.py, test_judge.py, test_monte_carlo.py, test_engine.py, test_cli.py, test_models.py, test_parser.py, test_reporter.py, test_corpus.py, test_elo.py, test_stats.py, test_harness_portability.py, test_e2e.py, conftest.py, test_e2e.py). End-to-end tests (`test_e2e.py`) run against real plugins. The judge layer generates 10 synthetic should-trigger / should-not-trigger prompts to compute triggering accuracy F1, which is a structural form of should-trigger regression. The Monte Carlo layer generates 15 varied prompts and runs N simulations for activation-rate statistics. [source](https://github.com/wshobson/agents/tree/main/plugins/plugin-eval/tests) and [source](https://github.com/wshobson/agents/blob/main/docs/plugin-eval.md) (Confidence: High)

11. **Security posture:** no dedicated secret-scan or prompt-injection mitigation found in plugin-eval's own tooling. The `validate.yml` CI does not include a gitleaks or trufflehog step. The `harness_portability` static check detects Claude-specific tool references and bare model aliases, but these are portability issues, not security posture checks. Installation is via `uv sync` (not curl-pipe-bash). No sandbox isolation for the Monte Carlo simulation runs. [source](https://github.com/wshobson/agents/blob/main/.github/workflows/validate.yml) (Confidence: High)

12. **Output formats:** JSON + Markdown + HTML. The `Reporter` class in `reporter.py` emits all three via `--output json|markdown|html` CLI flag. The `eval-report.yml` CI workflow writes per-plugin JSON files and a `summary.md` to a `eval-reports/` artifact directory, and posts `summary.md` to the GitHub Actions job summary. [source](https://github.com/wshobson/agents/blob/main/docs/plugin-eval.md) and [source](https://github.com/wshobson/agents/blob/main/.github/workflows/eval-report.yml) (Confidence: High)

13. **Packaging / install:** Python package via uv (`uv sync` in `plugins/plugin-eval/`). No npm/npx, no standalone GH Action, no global CLI install outside the monorepo. The plugin-eval package ships as part of the wshobson/agents monorepo and is consumed by cloning the repo and running `uv sync` in the subdirectory. Claude Code users install via `/plugin marketplace add wshobson/agents` and then access `/eval`, `/certify`, `/compare` slash commands. [source](https://github.com/wshobson/agents/blob/main/plugins/plugin-eval/README.md) and [source](https://github.com/wshobson/agents/blob/main/plugins/plugin-eval/pyproject.toml) (Confidence: High)

16. **Provenance taxonomy (separates portable-objective from house conventions?):** partial separation via the harness-portability sub-check. The static layer has seven sub-checks; six are generic quality heuristics (frontmatter quality, orchestration wiring, progressive disclosure, structural completeness, token efficiency, ecosystem coherence) and one is explicitly labeled harness portability (6% weight), which flags Claude-specific tool references and Codex-specific constraints. This partitions one class of house-convention findings into its own sub-check with its own score. However, the six generic sub-checks still embed heuristics specific to how Claude Code uses SKILL.md (e.g. trigger-phrase pattern, MUST/NEVER/ALWAYS density) without labeling them as Claude-specific vs portable-objective. No provenance field in the output JSON distinguishes portable spec failures from house-convention findings. ## Notes

- **Per-unit vs. whole-library dispute resolved.** Prior secondary claims that plugin-eval is "per-unit not library" are confirmed as accurate with precision: the _full_ three-layer evaluation is per-skill (SKILL.md directories), and plugin-level evaluation exists but runs only Layer 1 (static), with the confidence label forced to "Estimated" regardless of requested depth. The CLI source contains an explicit warning: "plugin-level evaluation only runs the static layer; judge and Monte Carlo layers require per-skill evaluation." The secondary claims were directionally correct but understated the nuance that a static-only plugin-level mode exists.

- **Three layers are a score pipeline, not a climbable tier system.** The Depth enum (quick/standard/deep/thorough) controls which layers run, and the confidence label (Estimated/Assessed/Certified/Certified+) reflects which layers contributed. This is not a tiered gate where a skill must pass Layer 1 before being allowed to proceed to Layer 2; all requested layers always run and their scores are blended. A skill does not "unlock" a higher tier by clearing lower ones.

- **javascript-typescript reference files verified intact (as of 2026-06-10).** The prior corpus run flagged BROKEN references/*.md links in the javascript-typescript plugin. Direct inspection of all four skills in that plugin confirms: every link in every SKILL.md that points to a file in `references/` resolves to an actual file present in the repo. `javascript-testing-patterns/SKILL.md` links to `references/advanced-testing-patterns.md` (file exists); `modern-javascript-patterns/SKILL.md` links to `references/details.md` and `references/advanced-patterns.md` (both exist); `nodejs-backend-patterns/SKILL.md` links to `references/details.md` (exists); `typescript-advanced-types/SKILL.md` references `references/details.md` (exists). The prior corpus finding of broken links is NOT reproduced by direct API inspection; the links appear to have been repaired since that run, or the prior finding was an error. This is an adjudication-eligible discrepancy.

- **No GitHub releases.** The repo has zero GitHub releases (releases API returns empty). Versioning is handled by commit + dependabot bumps only. The plugin-eval package carries version 0.1.0 in pyproject.toml but has never been published to PyPI or tagged as a GitHub release.

- **eval-report.yml runs weekly but does not gate merges.** The CI sweep grades all plugins on a Monday schedule and on manual dispatch. It uploads results as artifacts and posts to the job summary. It is not wired as a required status check; a PR can merge even if the weekly eval run reveals score regressions.

- **Copilot is a sixth harness.** The README badge list and validate.yml `multi-harness-generate` job list Copilot alongside Codex/Cursor/OpenCode/Gemini. The Makefile `HARNESSES := codex copilot cursor gemini opencode` confirms five generated harnesses. The README says "six harnesses" (Claude Code native + five generated). Prior secondary material that cited four or five harnesses understated the count.

- **LLM calls require `claude-agent-sdk` (Claude Code Max plan default) or `ANTHROPIC_API_KEY`.** The judge and Monte Carlo layers call Claude via `claude-agent-sdk` under the Max plan, or via the Anthropic SDK with an API key. The static layer has no LLM dependency and runs free.
