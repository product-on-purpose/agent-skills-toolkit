# skills-ref - profile

skills-ref is the reference validation library shipped inside the `agentskills/agentskills` repository (the GitHub org `agentskills`, owner type Organization). The Agent Skills format it validates "was originally developed by Anthropic, released as an open standard, and has been adopted by a growing number of agent products" (repo README). skills-ref is a small Python CLI (`pip install -e .` from the `skills-ref/` directory) exposing three commands - `validate`, `read-properties`, and `to-prompt` - that respectively check a skill directory against the spec, emit a skill's parsed properties as JSON, and render an `<available_skills>` XML block for agent prompts. Its own README carries an explicit disclaimer: "This library is intended for demonstration purposes only. It is not meant to be used in production." It is the canonical reference implementation of the agentskills.io base specification (the floor that this project's tiered Standard builds upon), which is why it is the baseline profile in this comparison. The repo code is Apache-2.0; documentation is CC-BY-4.0; the LICENSE copyright line reads "Copyright 2025 Anthropic, PBC".

## Dimensions

1. **Unit of evaluation:** per-skill. Every command takes a single skill directory path (`skills-ref validate path/to/skill`); `validate` resolves the path, requires a `SKILL.md`, and checks that one skill's frontmatter and directory name. There is no plugin-level or whole-library/catalog grading. (`to-prompt` accepts multiple skill paths but only to concatenate their prompt blocks, not to grade a collection.) [source](https://raw.githubusercontent.com/agentskills/agentskills/main/skills-ref/src/skills_ref/validator.py) (Confidence: High)
2. **Tiered and climbable:** pass-fail. `validate` collects a flat list of problems and reports pass-or-fail for the one skill; there are no tiers, no score, no burndown ladder, and no notion of climbing levels. The spec it enforces is a single flat conformance bar. [source](https://raw.githubusercontent.com/agentskills/agentskills/main/skills-ref/src/skills_ref/validator.py) (Confidence: High)
3. **Verdict basis:** deterministic. The validator performs Unicode normalization, string/length checks, character-class checks, and filesystem path checks entirely in local Python with no LLM call and no network access. The verdict is a pure function of the skill's bytes on disk. [source](https://raw.githubusercontent.com/agentskills/agentskills/main/skills-ref/src/skills_ref/validator.py) (Confidence: High)
4. **Lifecycle coverage:** validate only. skills-ref covers the validate stage of the lifecycle. It does not author/scaffold skills, does not version or release them, and has no deprecate or govern function. (`read-properties` and `to-prompt` are read/transform utilities, not lifecycle stages.) [source](https://raw.githubusercontent.com/agentskills/agentskills/main/skills-ref/README.md) (Confidence: High)
5. **Cross-agent emission:** none. skills-ref does not emit or transpile skills to any agent-specific package format (no Claude plugin, no Codex `.codex-plugin/`, no Gemini format). Its only output-shaping command, `to-prompt`, produces a generic spec-defined `<available_skills>` XML block, not a per-vendor package. [source](https://raw.githubusercontent.com/agentskills/agentskills/main/skills-ref/README.md) (Confidence: High)
6. **Self-proving / dogfooded:** not as a published CI gate. The `skills-ref/` directory contains a `tests/` folder (unit tests for the library), but no primary source shows skills-ref running its own `validate` as a gate over the repository's own skills in CI; the repo's GitHub Actions / CI wiring for self-validation is not documented in the README. Treated as absent pending evidence. [source](https://github.com/agentskills/agentskills/tree/main/skills-ref) (Confidence: Medium - tests/ directory is present and confirmed; the absence of a self-gating CI workflow is absence of evidence in fetched sources, not a confirmed negative)
7. **Target spec(s):** agentskills.io base specification only. The validator enforces exactly the fields and constraints defined at agentskills.io/specification (name, description, optional license/compatibility/metadata/allowed-tools); it implements no Claude-extended, Codex, or Gemini superset. [source](https://agentskills.io/specification) (Confidence: High)
8. **Validation depth:** frontmatter-quality + structure (directory + required-file + closed field set), against the base spec. Confirmed rules in `validator.py`: skill path must exist and be a directory; `SKILL.md` required ("Missing required file: SKILL.md"); `name` must be lowercase, only letters/digits/hyphens, not start or end with a hyphen, no consecutive hyphens, max 64 chars, and must match the parent directory name; `description` must be a non-empty string, max 1024 chars; `compatibility` (if present) max 500 chars; only the spec's allowed fields are permitted ("Unexpected fields in frontmatter ... Only [allowed] are allowed"); `license`/`metadata`/`allowed-tools` are accepted but not deeply validated. It does NOT do reference/budget linting, manifest-vs-disk reconciliation, strict-CI gating, or semantic-quality scoring. The README is explicit that this is a demonstration/reference implementation, not a production linter (see dim 15 / Notes). [source](https://raw.githubusercontent.com/agentskills/agentskills/main/skills-ref/src/skills_ref/validator.py) (Confidence: High)
9. **Versioning support:** none enforced. The spec defines `metadata` as an arbitrary key-value map and shows `version: "1.0"` only as an example metadata key; skills-ref does not require, parse-with-meaning, or enforce semver, contract-locks, or provenance trails on a skill's version. Version is an unvalidated free-form metadata string. [source](https://agentskills.io/specification) (Confidence: High)
10. **Eval / test scaffolding:** none for skill behavior. skills-ref ships unit tests for the library itself (`skills-ref/tests/`), but provides no should-trigger harness, no `evals.json`, no regression suite, and no adversarial eval framework for the skills it validates. [source](https://github.com/agentskills/agentskills/tree/main/skills-ref) (Confidence: High)
11. **Security posture:** none. No secret scanning, no prompt-injection mitigation, no sandboxing, and no install-time code execution risk surface beyond a standard local `pip install -e .` (the disclaimer further warns against production use). The validator does not execute or fetch skill content; it only parses frontmatter and inspects the filesystem. [source](https://raw.githubusercontent.com/agentskills/agentskills/main/skills-ref/README.md) (Confidence: High)
12. **Output formats:** human (text problems from `validate`) + JSON (`read-properties` "outputs skill data as JSON") + XML (`to-prompt` emits an `<available_skills>` block). The README explicitly does not reference SARIF, GitHub annotations, HTML, or any CI-oriented output format. [source](https://raw.githubusercontent.com/agentskills/agentskills/main/skills-ref/README.md) (Confidence: High)
13. **Packaging / install:** manual local pip (editable install). Install is `python -m venv .venv`, activate, then `pip install -e .` from the `skills-ref/` directory (or `uv sync`); after setup the `skills-ref` executable is on PATH. The README documents no published PyPI release, no npx/CLI distribution, no marketplace, and no GitHub Action. (The package is named `skills-ref`, but the documented install path is the in-repo editable install.) [source](https://raw.githubusercontent.com/agentskills/agentskills/main/skills-ref/README.md) (Confidence: High)
14. **Maturity signal:** repository `agentskills/agentskills` snapshot 2026-06-10: 20,263 stars, 1,259 forks, 49 open issues; created 2025-12-16; last push 2026-05-20; default branch `main`; license Apache-2.0. These figures describe the whole spec repository (which the high star count reflects), not skills-ref in isolation; skills-ref has no separate release/tag series in the fetched sources. [source](https://api.github.com/repos/agentskills/agentskills) (Confidence: High-dated-snapshot)
15. **Governance / standing:** first-party reference implementation of an Anthropic-originated open standard; NOT foundation-governed as of 2026-06-10. Adjudication (this is the project's canonical governance ruling): the claim under test is "the Agent Skills specification is AAIF / Linux-Foundation-governed." Primary evidence: (a) the `agentskills/agentskills` LICENSE reads "Copyright 2025 Anthropic, PBC" (Apache-2.0 code, CC-BY-4.0 docs), and the README states the format "was originally developed by Anthropic, released as an open standard" - no foundation is named as steward or copyright holder; (b) the aaif.io hosted-project list names exactly four projects - Model Context Protocol (MCP), goose, AGENTS.md, and agentgateway - and does NOT list Agent Skills, even though it does list MCP, goose, and AGENTS.md. Ruling: the Agent Skills spec is NOT AAIF / Linux-Foundation-governed as of 2026-06-10; stewardship sits with Anthropic / the `agentskills` org. What is established: Anthropic is the copyright holder and originator; AAIF is a Linux Foundation project; AAIF does not host Agent Skills. What is NOT established (and was not claimed): that an AAIF donation is forthcoming or has been discussed - no primary source speaks to future intent either way. [source](https://raw.githubusercontent.com/agentskills/agentskills/main/LICENSE) and [source](https://aaif.io) (Confidence: High)
16. **Provenance taxonomy (separates portable-objective from house conventions?):** n/a (no house layer to separate). skills-ref validates only the agentskills.io base spec, so every rule it enforces is by construction a portable, objective spec rule; there are no house/editorial conventions mixed in, and therefore nothing for a provenance taxonomy to disambiguate. The distinction this dimension probes does not arise for a pure base-spec validator. [source](https://raw.githubusercontent.com/agentskills/agentskills/main/skills-ref/src/skills_ref/validator.py) (Confidence: High)

## Notes

- **AAIF / Linux Foundation adjudication (canonical record).** This profile carries the project's deciding governance ruling, pre-registered in METHODOLOGY.md as the worked example. The conflict: some secondary reports asserted the Agent Skills spec is AAIF / Linux-Foundation-governed; a careful report said it was not (as of mid-2026); and this project's own prior positioning notes once asserted "governance -> Linux Foundation." Resolved from primary sources only. (a) `agentskills/agentskills/LICENSE`: "Copyright 2025 Anthropic, PBC"; README: the format "was originally developed by Anthropic, released as an open standard." No foundation is named as owner or steward. (b) `aaif.io` lists exactly MCP, goose, AGENTS.md, and agentgateway as its projects; Agent Skills is absent, while three comparator projects (MCP, goose, AGENTS.md) are present - so the absence is a deliberate non-listing, not an incomplete page. Verified 2026-06-10. RULING: the Agent Skills specification is NOT AAIF / Linux-Foundation-governed as of 2026-06-10; it is a first-party Anthropic-originated open standard published under the `agentskills` org. The earlier "governance -> Linux Foundation" positioning is corrected by this record. Adjudication-log row for METHODOLOGY.md: | Agent Skills spec is AAIF-governed | Gemini/ChatGPT reports (yes) vs careful Claude report + prior positioning note (no) | `agentskills/agentskills` LICENSE ("Copyright 2025 Anthropic, PBC") + README; `aaif.io` project list (MCP, goose, AGENTS.md, agentgateway - no Agent Skills) | NOT AAIF-governed; Anthropic-stewarded open standard | 2026-06-10 |.
- **Distinguish the validator from the spec.** Two different artifacts share the `agentskills/agentskills` repo. The SPEC (agentskills.io/specification) is the governed-or-not object the adjudication is about. The VALIDATOR (skills-ref) is a Python CLI inside that repo. The governance ruling above is about the spec's stewardship; skills-ref itself is unambiguously a first-party reference tool.
- **"Demonstration purposes only."** The skills-ref README states verbatim: "This library is intended for demonstration purposes only. It is not meant to be used in production." This is load-bearing for the comparison: skills-ref defines the conformance floor (what a skill must minimally satisfy) but explicitly disclaims being a production-grade quality gate. The base spec is a floor; the production tiering/burden layer is exactly the gap a tiered Standard fills.
- **Why this is the baseline.** Because skills-ref is the reference implementation of the agentskills.io base spec, its rule set is the floor every other tool in the matrix sits above. The 16 dimensions here read mostly as "single skill / pass-fail / deterministic / base-spec-only / no eval / no security / no emission / no versioning" by design - that is the definition of the floor, not a deficiency relative to its stated purpose.
- **Stars belong to the repo, not the package.** The 20,263-star count is for the whole specification-and-docs repository; it should not be read as adoption of skills-ref the library specifically. skills-ref has no independent release series in the fetched sources.
- **No CI / SARIF / annotations.** Confirmed absent in the skills-ref README: no GitHub Actions wiring, no SARIF, no GitHub annotations, no HTML. Output is text (validate), JSON (read-properties), and XML (to-prompt) only.
- **Optional fields lightly handled.** `license`, `metadata`, and `allowed-tools` are accepted as allowed fields but are not deeply validated; `allowed-tools` is flagged Experimental in the spec.

Library unit tests under `skills-ref/tests/` are confirmed present, but self-validation-as-gate is unverified.

The repo's own documented install path remains the in-repo editable `pip install -e .` (recorded in dimension 13), but the PyPI wheel exists at v0.1.1.
