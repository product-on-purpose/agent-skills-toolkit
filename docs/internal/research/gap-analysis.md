# How to stay best in class: gap analysis

This is the ranked worklist derived from `matrix.md` (the askit row vs nine competitors across 16 sourced dimensions): what to adopt from the field, what to build to widen the moat, and what to deliberately skip because it conflicts with askit's identity. Every item cites the `tool-profiles/<tool>.md` that motivates it. The lens throughout is askit's identity: a deterministic (model-free) gate that grades a whole library against a climbable tier with a burndown, emits cross-agent from one library.json, proves itself at Gold in CI, and splits its checks by provenance (objective / vendor-cited / house, ADR 0029). Judgment-based evaluation exists, but it sits beside the gate as opt-in evidence and never decides pass or fail.

## Adopt (clear wins to take)

These are capabilities the field has shipped that fit askit's identity cleanly. Ordered best-first.

### 1. SARIF + GitHub-annotation output - effort S

- **What:** A SARIF 2.1.0 report writer and `::error`/`::warning` GitHub Actions annotations, in addition to the existing human, HTML, and Markdown outputs.
- **Who has it:** skill-check emits text, JSON, SARIF, HTML, and GitHub annotations, all five enumerated from source (`tool-profiles/skill-check.md` dim 12; SARIF via `src/core/sarif.ts`, annotations via `src/core/github-formatter.ts`). skills-check emits SARIF 2.1.0 specifically "for GitHub Security tab" plus Markdown and JSON (`tool-profiles/skills-check.md` dim 12). The matrix records askit as "human + HTML + Markdown report; no SARIF" (dim 12), so this is the one place a single-vendor and a Vercel-affiliated tool both out-cover us on a pure-presentation axis.
- **Why it fits:** askit's findings are already deterministic and structured; SARIF and annotations are a new serialization of data the gate already computes. Nothing about the verdict, the tiers, or the burndown changes. It is CI-native (SARIF lands findings in the GitHub Security tab; annotations inline them on the PR diff), which reinforces the "runs as a gate in CI" identity rather than diluting it. It is also the natural surface for the provenance split: each SARIF result carries a `ruleId`, and the objective/vendor-cited/house class can ride along as a rule property (see Build item 3).
- **Effort:** S. One new emitter that maps the existing findings model to the SARIF schema, plus a thin annotations formatter. No new checks, no gate logic, deterministic.

### 2. semver-bump-vs-content-diff verification - effort M

- **What:** Given a declared version bump in a skill's metadata and the actual content diff against the prior version, verify that the change magnitude justifies the declared bump (a patch that rewrites behavior, or a major that fixes a typo, is flagged).
- **Who has it:** skills-check's `verify` is the standout: it reads the declared version from frontmatter, retrieves the prior version from git, computes the declared bump level by semver comparison, then runs three parallel heuristic analyses (structural diff, package diff, content similarity) and flags a mismatch as a failure - with a `--skip-llm` flag that keeps the whole path deterministic (`tool-profiles/skills-check.md` dim 9, and Notes: "verify is the standout dimension ... the `--skip-llm` fallback ensures it remains usable without API keys"). The matrix records askit as "semver-enforced + provenance" (dim 9): askit enforces that a version exists and is valid semver, but does not verify a declared bump against the diff. This closes a real, named gap against the broadest competitor.
- **Why it fits:** the deterministic heuristic-only path (structural + content-similarity diff, no LLM) is exactly the kind of check askit's gate can own. It extends the versioning dimension askit already leads on (askit is one of the few that pairs semver with provenance) without reaching for a model in the verdict. The LLM-classification half of skills-check's design is the part askit should leave out of the gate (it can live as opt-in evidence; see Skip).
- **Effort:** M. Requires a prior-version retrieval step (git history or an explicit before/after), a diff-magnitude heuristic, and a mapping from heuristic magnitude to expected bump level. The whole thing stays deterministic if the LLM path is omitted.

### 3. Prompt-injection + curl-pipe-bash scan - effort M

- **What:** Extend askit's security checks beyond secret-scan to cover (a) `curl|bash` / `wget|sh` pipe-to-shell patterns (including process-substitution variants) in prose and code blocks, and (b) prompt-injection / dangerous-command patterns.
- **Who has it:** skills-validator detects curl/wget-piped-to-shell via regex in both prose and code blocks at Warning severity, plus process-substitution variants like `bash <(curl ...)`, with optional semgrep rule sets - all deterministic (`tool-profiles/skills-validator.md` dim 11). skills-check's `audit` scans prompt-injection patterns, dangerous shell commands (`rm -rf`, `curl | bash`), and hallucinated packages (`tool-profiles/skills-check.md` dim 11). skill-check delegates security to the external `mcp-scan` tool through its `agent-scan` adapter, which the profile flags as a real operational gap because mcp-scan is not bundled and the Action defaults the scan to false (`tool-profiles/skill-check.md` dim 11 and Notes). The matrix records askit as "secret-scan (U11) + least-privilege settings; no curl-pipe-bash scan" (dim 11). A library grader that ships skills for execution should catch a pipe-to-shell installer.
- **Why it fits:** these are pattern-match (regex / static) checks, fully deterministic, and they belong on the objective tier (they are portable, vendor-neutral hazards, not house style). They strengthen the gate without an LLM. Borrowing skills-validator's specific regex set (pipe-to-shell + process substitution) is the highest-value, lowest-risk slice; the hallucinated-package half (which needs live npm/PyPI lookups and caching, per skills-check Notes) is a heavier, separately-scoped follow-on.
- **Effort:** M. The curl-pipe-bash regexes alone are S; adding a curated prompt-injection / dangerous-command pattern list and wiring severity onto the right provenance tier pushes it to M. Bundle the patterns rather than shelling to an external tool, so askit avoids skill-check's "external dependency not in our release cycle" gap.

### 4. Deterministic `--fix` / doctor auto-remediation - effort M

- **What:** A `--fix` mode that deterministically repairs the mechanically-fixable subset of findings in place (trailing newline, field ordering, simple frontmatter normalization), leaving judgment findings untouched.
- **Who has it:** skill-check applies deterministic auto-fixes via `--fix` (`tool-profiles/skill-check.md` intro and Notes, and `src/core/fix.ts`); the profile notes the fix path writes files with `fs.writeFileSync()` and flags that the documented `--fix --interactive` mode is not clearly implemented in source (an open item, so we should ship non-interactive and explicit). skills-check has `lint --fix` (fills missing frontmatter from git context) and a `doctor` release-readiness command (`tool-profiles/skills-check.md` dim 4). askit reports findings but does not auto-fix.
- **Why it fits:** the fixes are deterministic transforms of deterministic findings, so they stay inside the model-free identity. This is genuinely useful for the burndown: it shortens the climb from Bronze toward Gold by clearing the mechanical findings automatically, which is on-brand for a climbable tier with a burndown. Keep it conservative - only fix findings with a single unambiguous correct form, never anything judgment-based.
- **Effort:** M. Needs a per-check "fixable" capability flag, a safe in-place writer, and tests proving idempotence (fix-then-recheck is clean). Scope it to the unambiguous findings first.

### 5. Deeper eval / regression harness - selective borrow, effort M

- **What:** Targeted hardening of askit's existing opt-in behavioral mode and G3 regression check, borrowing specific mechanisms rather than adopting a competitor's whole harness.
- **Who has it:** skill-creator is the reference design - a 60/40 train/test split optimization loop (`run_loop.py`) that selects the best description by held-out test score to prevent overfitting, parallel with-skill vs baseline runs, and an HTML eval-viewer with a `--static` headless mode (`tool-profiles/skill-creator.md` dim 10). skills-check's `test` runs eval suites with trigger/outcome/style/regression test types, multi-trial execution with pass thresholds, baseline regression detection, and cost caps - but it runs arbitrary shell via agent harnesses and the profile flags it as requiring container isolation (`tool-profiles/skills-check.md` dim 10 and Notes). The matrix records askit as "regression + behavioral (opt-in LLM-judge)" (dim 10), so askit already has the spine; this is about what to borrow, not whether to build.
- **Why it fits, and the boundary:** the borrowable, identity-safe pieces are the deterministic scaffolding around evaluation: held-out test/train splitting to prevent overfitting (skill-creator), baseline-diff regression with explicit accept/update (skills-check), multi-trial with a pass threshold, and cost caps. These harden askit's opt-in evidence layer without moving the LLM into the verdict. The pieces to leave out are anything that makes a model the grader of record (that stays beside the gate) and skills-check's unsandboxed shell execution (only adopt with the isolation boundary it documents).
- **Effort:** M for the harness hardening (split logic, baseline-diff ergonomics, trial/threshold config). Treat full sandboxed behavioral execution as a larger, separately-scoped effort.

## Build (differentiated, worth building ourselves)

No competitor combines whole-library + tiered-with-burndown + deterministic + cross-agent + self-proving + provenance (matrix "What the matrix shows": "only agent-skills-toolkit combines all six structural traits at once"). The Build items extend that combination into things the field structurally cannot copy cheaply, because each leans on a trait the competitors lack.

### 1. A published conformance suite others can run - effort M

- **What:** Package askit's objective-tier checks (and a fixture corpus of pass/fail skills) as a portable conformance suite that any author or competing tool can run against a library to get the same deterministic verdict askit produces - the standard made executable and externally reproducible.
- **Why askit and not the field:** askit is the only tool that is both deterministic and self-proving in CI (matrix dims 3 and 6; askit "yes (self-validates at Gold in CI)"). A deterministic, self-proving grader is the only one that can credibly publish "run this and you will get the same answer we get." skills-ref is the base-spec floor but is self-described as for demonstration only (matrix "What the matrix shows"), and skill-check / skills-check / skills-validator are not confirmed to run their own checks against themselves in CI (`tool-profiles/skill-check.md` dim 6 "No"; `tool-profiles/skills-check.md` dim 6 "not confirmed ... self-check invocation is absent"; `tool-profiles/skills-validator.md` dim 6 "Not confirmed ... no SKILL.md exists in the repository"). That gap is the opening: askit can offer reproducible conformance as a contract precisely because it already holds itself to it.
- **Effort:** M. The checks and fixtures largely exist; the work is packaging them as a standalone, versioned, runnable suite with a stable report contract.

### 2. Make the provenance split a consumable contract - effort S

- **What:** Expose the objective / vendor-cited / house classification (ADR 0029) as first-class machine-readable output on every finding - in JSON, and as a SARIF rule property once Adopt item 1 lands - and document it as a stable contract so a consumer can filter to "portable objective failures only."
- **Why askit and not the field:** the provenance taxonomy is the single dimension where askit stands alone. The matrix dim 16 reads askit "yes (objective/vendor-cited/house, ADR 0029)" against a field of "no" (skill-check, ccpi, skill-creator), "partial" (skills-check, plugin-eval, skills-validator), and "n/a." The closest competitors only separate concerns structurally - skills-check by different commands (`audit` vs `policy`), plugin-eval by one `harness_portability` sub-check, skills-validator by severity convention - and none attaches a provenance label to each finding in its output schema (`tool-profiles/skills-check.md` dim 16; `tool-profiles/plugin-eval.md` dim 16; `tool-profiles/skills-validator.md` dim 16). Turning askit's split from an internal taxonomy into a documented output contract makes the differentiator something downstream tooling can build on - the thing that lets a third party grade a library they do not own on portable rules only, which is exactly the third-party-grading use case ADR 0029 was written for.
- **Effort:** S. The classification already exists per check; this is surfacing it in the output schema and writing the contract down.

### 3. Forward-looking MCP-served-skill validation - effort L, speculative

- **What:** Validation for skills delivered over MCP (served by a server) rather than as files on disk - schema, provenance, and budget checks that work when the "library" is a set of MCP-exposed capabilities.
- **Why askit and not the field:** this is genuinely speculative and marked as such. No tool in the corpus validates MCP-served skills; every profiled tool operates on on-disk SKILL.md files (the unit-of-evaluation row across the matrix is uniformly file-based). askit's whole-library framing (matrix dim 1, the only "whole-library" entry) is the closest conceptual starting point because it already reasons about a collection rather than a single file. Flagged speculative: the MCP-served-skill distribution model may not converge, so this is a watch-and-prototype item, not a committed build. Revisit when the eval-target corpus surfaces a real MCP-served library to test against.
- **Effort:** L, and contingent on the ecosystem moving. Do not start until there is a concrete target.

## Deliberately skip (and why)

Capabilities the field has that askit should not adopt, because they conflict with its identity.

### 1. Per-skill LLM-judge or Monte Carlo as the gate verdict - skip

- **What it is:** plugin-eval blends a static layer with an LLM judge (four async Claude calls scoring triggering accuracy, orchestration, output quality, scope) and a Monte Carlo layer (50-100 LLM calls for activation-rate statistics) into a single composite 0-100 score that drives the badge tier (`tool-profiles/plugin-eval.md` dim 3 and dim 10).
- **Why skip:** this is the direct antithesis of askit's identity. askit's verdict is deterministic and model-free (matrix dim 3: askit "deterministic" vs plugin-eval "hybrid (3-layer + LLM judge)"); a non-reproducible, paid, latency-bound LLM score as the pass/fail authority would break reproducibility, CI determinism, and the self-proving guarantee. askit already keeps the useful half - it has an opt-in behavioral / LLM-judge mode (matrix dim 10) - but it stays beside the gate as evidence and never decides the verdict. Note also that even ccpi, the closest comparator, keeps its LLM path "advisory-only ... never drives the merge verdict" (`tool-profiles/ccpi.md` dim 3); making an LLM the gate is the line none of the deterministic-minded tools cross.

### 2. A competing distribution / package-manager CLI - skip

- **What it is:** a CLI that installs, updates, and removes skills across agents - the Vercel `skills` CLI is the de-facto cross-agent package manager (matrix dim 13 and dim 5: "multi-format native (80 agents)"), and ccpi ships `ccpi install <name>` over a 425-447 plugin catalog (`tool-profiles/ccpi.md` dim 13).
- **Why skip:** askit already installs via the Claude Code marketplace plus a plugin bundle (matrix dim 13), so building a competing installer is undifferentiated work against entrenched incumbents - the Vercel CLI sits on a repo with ~22k stars and 80-agent reach (matrix dim 14, dim 5), and distribution is a gravity well, not a grading advantage. It would also blur askit's identity from "the grader/standard" into "yet another package manager," diluting the one thing that is defensible. Emission (one library.json to Claude Code + Codex) is in scope; distribution mechanics are not.

### 3. Pure scaffolding-only features - skip

- **What it is:** authoring-only scaffolding with no grading - skill-creator is the reference (an authoring meta-skill: intent capture, drafting, packaging to `.skill`, with no tiers, no score, no verdict; `tool-profiles/skill-creator.md` dim 2 "None" and dim 3 "Not applicable"). skill-check's `new` template scaffold and skills-check's `init` are lighter versions of the same (`tool-profiles/skill-check.md` dim 10; `tool-profiles/skills-check.md` dim 4).
- **Why skip:** scaffolding is commoditized - the official first-party tool (skill-creator) already owns it, and askit's value is the gate and the standard, not generating boilerplate. There is one nuance worth keeping separate: borrowing skill-creator's deterministic *eval-loop mechanics* (held-out split, benchmarking) is in scope under Adopt item 5; what askit should skip is building a general-purpose authoring/scaffolding surface that competes with skill-creator on its own ground.

## Honesty notes

- **Do not claim description-quality superiority yet.** askit's U5 description scorer clusters good `use-when` descriptions at ~0.65, under its own 0.7 bar, and likely saturates; recalibration is pending (DESIGN 3b). Until that is fixed, askit must not assert it grades description quality better than the field - several competitors run description rules too (skill-check has four description rules including min/max length and "Use when" phrasing, `tool-profiles/skill-check.md` dim 8), and an uncalibrated scorer is not a credible basis for a superiority claim. ADR 0029 already made U5 a house check, so it does not affect outward third-party grades, but the calibration caveat still gates any public "better descriptions" messaging.
- **The provenance taxonomy is a real, sole-held differentiator - surface it.** The objective / vendor-cited / house split (ADR 0029) is matched by no competitor: matrix dim 16 shows askit "yes" against the field's "no" / "partial" / "n/a." The nearest attempts separate concerns only structurally (skills-check by command, plugin-eval by one sub-check, skills-validator by severity) and none labels each finding with its provenance in output (dims 16 across those profiles). This is the cleanest thing to lead with, and Build item 2 turns it from an internal property into a consumable contract.
